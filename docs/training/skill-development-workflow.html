一、背景与问题

2026-03-01 的协作中，遇到了以下核心问题：

问题 1：多 Agent 配置反复试错
尝试 5+ 种方案（WebSocket、独立 Gateway、Profile 模式），均因环境限制失败，浪费大量时间。
问题 2：视觉模型配置无效
修改本地配置文件 3 次，均被系统自动重置，未预先验证配置是否可持久化。
问题 3：流程执行不稳定
标准流程（需求改写→执行计划→记忆记录）时有时无，导致协作体验不一致。

二、核心洞察

洞察 1：先验证，再执行
任何配置修改前，先验证其持久性。OpenClaw 的模型配置由服务端控制，本地修改无效。
洞察 2：技能优先于试错
今日所有问题都有对应的 skill 可解决（skill-creator、channels-setup），应先学习 skill 而非盲目尝试。
洞察 3：流程是纪律，不是选项
标准流程不是"可选项"，而是保障质量的机制。无论任务大小、时间多晚，必须执行。

三、标准化技能开发流程

1
抽象关键任务

从今日工作中提取核心痛点，明确需要解决什么问题。

示例：多 Agent 隔离失败 → 需要学习正确的 agent 管理方法

2
ClawHub 搜索技能

使用 openclaw skills search 或 npx clawhub 搜索相关技能。

关键技能：

  • skill-creator - 创建和管理技能
  • channels-setup - 配置 IM 渠道
  • clawhub - 技能市场管理
3
评估学习价值

判断技能是否能解决当前问题，并带来长期价值。

评估维度 问题 价值
紧迫性 是否阻塞当前任务？ 高
复用性 未来是否会再次遇到？ 高
深度 是否能提升核心能力？ 中
4
学习技能并实践

阅读 SKILL.md，理解核心原理，立即应用于当前任务。

skill-creator/ ├── SKILL.md # 核心文档（必须阅读） ├── scripts/ │ ├── init_skill.py # 初始化技能模板 │ └── package_skill.py # 打包技能 └── references/ ├── workflows.md # 多步骤流程设计 └── output-patterns.md # 输出格式规范
5
生成能力教材

将学习成果转化为可复用的 HTML 报告，包含：

  • 背景与问题（真实案例）
  • 核心洞察（抽象规律）
  • 标准化流程（可执行步骤）
  • 对比/效果（改进前后）
6
发布与推广

部署到网站，推送钉钉，形成知识沉淀。

发布 checklist：

  • ✅ GitHub Pages 部署
  • ✅ 巡检验证（HTTP 200）
  • ✅ 钉钉推送（封面+标题+URL）
  • ✅ 更新 MEMORY.md

四、技能目录结构规范

基于 skill-creator 最佳实践，标准技能结构如下：

my-skill/ ├── SKILL.md # 必需：技能说明文档 │ ├── YAML frontmatter # name + description（触发器） │ └── Markdown body # 使用说明（触发后加载） ├── scripts/ # 可选：可执行脚本 │ └── example.py ├── references/ # 可选：参考资料 │ └── details.md └── assets/ # 可选：模板资源 └── template.html

关键原则

  • 简洁优先：SKILL.md 控制在 500 行以内
  • 渐进披露：元数据 → SKILL.md → 参考资料
  • 不要冗余：不创建 README.md、CHANGELOG.md 等

五、效果对比

维度 改进前 改进后
试错次数 5+ 次 1 次（按 skill 执行）
配置持久性 反复被重置 先验证再执行
流程稳定性 时有时无 100% 执行
知识沉淀 无 HTML 教材 + 钉钉推送
